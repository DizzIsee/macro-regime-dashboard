## 1. Project Overview

A Bloomberg Terminal-style **Macro Regime Dashboard** originally generated by Perplexity AI as a static HTML prototype. The goal is to transform it into a fully self-sustaining web app that:

- Pulls **100% live data** automatically (no manual updates ever)
- Costs **~$0/month** to run
- Deploys on **Vercel via GitHub** (`DizzIsee` GitHub account)
- Updates daily via **GitHub Actions** cron job
- Preserves the **exact existing UI** (dark terminal theme, Bloomberg aesthetic)

The three source files are in this repo:
- `index.html` — full dashboard UI (1,374 lines)
- `style.css` — dark terminal theme (1,740 lines)
- `data/chart_data.json` — historical liquidity + SPX data (2016–Mar 2026, weekly)

## 2. The 8 Flaws to Fix (in priority order)

### 🔴 FLAW 1 — All data is hardcoded static strings
**Problem:** Every number in the dashboard (CPI, NFP, VIX, SPX, FedWatch probabilities, liquidity scores) is a string Perplexity wrote on April 1, 2026. Nothing updates. The "LIVE" badge is cosmetic only.

**Fix:** Build a Python data pipeline (`scripts/fetch_data.py`) that fetches all data from FRED API + Yahoo Finance and writes two JSON files: `data/macro_data.json` and `data/chart_data.json`. The browser JS then reads these JSON files on page load and populates every card, table, and chart dynamically. Zero hardcoded data values remain in `index.html`.

---

### 🔴 FLAW 2 — Liquidity chart not wired to data
**Problem:** The `chart_data.json` file exists and has good data (weekly, 2016–Mar 2026) but the chart JS reads from a hardcoded inline variable, not from the file. The chart also stops at March 25, 2026 — needs to stay current.

**Fix:** 
1. The Python script extends `chart_data.json` by fetching new weeks from FRED (WALCL, WTREGEN, RRPONTSYD, SP500 series) and appending them.
2. The chart JS does `fetch('./data/chart_data.json')` on load and renders from the response.
3. The liquidity score formula: `us_score = normalize(WALCL - WTREGEN - RRPONTSYD)` where normalize = percentile rank over the full 10Y window. Global score = weighted average of Fed + ECB + BOJ + PBOC balance sheets normalized similarly. Composite = average of US and Global scores.

---

### 🔴 FLAW 3 — T5YIE breakeven chart is broken (empty placeholder)
**Problem:** The chart JS contains the literal string `'[INSERT HISTORICAL T5YIE ARRAY HERE]'` as a placeholder. The chart renders as a blank box with a warning message.

**Fix:** The Python script fetches `T5YIE` series from FRED API (daily, going back 5 years) and writes it to `data/t5yie_data.json` as `[{"date": "YYYY-MM-DD", "value": 2.54}, ...]`. The chart JS does `fetch('./data/t5yie_data.json')` on load. The Fed 2% target dashed overlay is already pre-configured in the JS — just needs real data.

**FRED Series ID:** `T5YIE`

---

### 🔴 FLAW 4 — FedWatch probabilities are hardcoded and can't be scraped
**Problem:** CME FedWatch is JS-rendered and bot-protected. The dashboard shows a full probability table (97.4% hold at Apr 29, etc.) that Perplexity hardcoded. These change daily.

**Fix:** Derive FedWatch-equivalent probabilities from **30-Day Fed Funds Futures prices** using the same methodology CME uses:

```
Implied Rate = 100 - Futures Price
Probability of cut = (Current Rate - Implied Rate) / 0.25
```

Yahoo Finance tickers for Fed Funds Futures (ZQ contracts):
- `ZQK26.CBT` = May 2026 (covers Apr 29 FOMC)
- `ZQM26.CBT` = Jun 2026 (covers Jun 17 FOMC)
- `ZQN26.CBT` = Jul 2026 (covers Jul 29 FOMC)
- `ZQU26.CBT` = Sep 2026 (covers Sep 16 FOMC)
- `ZQV26.CBT` = Oct 2026 (covers Oct 28 FOMC)
- `ZQZ26.CBT` = Dec 2026 (covers Dec 9 FOMC)

The Python script fetches these via `yfinance`, calculates probabilities, and writes to `data/fomc_probs.json`. The FOMC probability table in the UI reads from this JSON.

**Important:** Verify the ZQ contract tickers are returning valid data before wiring to UI. If a ticker is stale or delisted, flag it clearly in the script output.

---

### 🔴 FLAW 5 — Signal Distribution gauge is manually scored, not algorithmic
**Problem:** The hawk/dove/tight/neutral score (3/2/3/7 = 15 points) is a hand-crafted judgment. When data updates, scores won't change.

**Fix:** Implement the scoring algorithm in the Python script based on the rules already visible in the HTML:

```python
# HAWKISH signals (inflation running hot)
hawk = 0
if ppi_yoy > ppi_est: hawk += 1          # PPI beat
if core_pce_yoy > core_pce_est: hawk += 1 # Core PCE beat
if fed_action == "hold": hawk += 1         # Fed on hold despite inflation

# DOVISH signals (labor weakening)
dove = 0
if unemp_rate > unemp_est: dove += 1      # Unemployment above estimate
if nfp < 0 or nfp < nfp_est: dove += 1   # NFP miss

# TIGHTENING signals (financial conditions)
tight = 0
if yield_10y > 4.0: tight += 1           # 10Y above 4%
if wti > 80: tight += 1                   # WTI above $80
if vix > 20: tight += 1                   # VIX elevated

# NEUTRAL = 15 - hawk - dove - tight
neutral = 15 - hawk - dove - tight
```

Scores are written to `data/macro_data.json` and the gauge bar `flex` widths + counts are set dynamically by JS.

### 🟡 FLAW 6 — Bottom Line synthesis text is hardcoded AI prose
**Problem:** The macro narrative paragraph ("Fed & Monetary:", "Labor & Data:", etc.) is from April 1, 2026 and never updates.

**Fix (two options — implement Option A first):**

**Option A (free):** Write a JS function that auto-generates templated narrative from the live data JSON. Not as rich as AI prose but always accurate and always current. Example: *"The Federal Reserve held at {fed_rate} for the {nth} consecutive meeting. Core PCE at {core_pce}% remains {above/below} the 2% target..."*

**Option B (~$0.05/day):** GitHub Action calls Claude API (claude-sonnet-4-6) once per day, passes the fresh `macro_data.json` as context, and generates the narrative paragraphs. Output saved to `data/synthesis.json`. UI reads and renders it. Add this only after Option A is confirmed working.

---

### 🟡 FLAW 7 — Perplexity inline-edit script is junk code
**Problem:** Lines 1311–1372 of `index.html` contain Perplexity's internal screenshot capture/iframe messaging script. It's their editor tooling — completely irrelevant to the deployed app. Dead weight.

**Fix:** Delete the entire `<script data-pplx-inline-edit>` block (lines 1311–1372). No replacement needed.

---

### 🟡 FLAW 8 — Header date/time is hardcoded
**Problem:** `APR 01, 2026 — 10:26 AM PDT` is a static string in the HTML `<time>` element.

**Fix:** Replace with a JS snippet that runs on page load:
```javascript
const now = new Date();
document.querySelector('.dash-header time').textContent = 
  now.toLocaleDateString('en-US', { month:'short', day:'2-digit', year:'numeric' }).toUpperCase() 
  + ' — ' + now.toLocaleTimeString('en-US', { hour:'2-digit', minute:'2-digit', timeZoneName:'short' });
```
Also update the `datetime` attribute dynamically.

## 7. GitHub Actions Workflow

```yaml
# .github/workflows/daily_update.yml
name: Daily Data Update

on:
  schedule:
    - cron: '0 13 * * 1-5'   # 6:00 AM PDT (13:00 UTC) Mon–Fri
    - cron: '0 14 * * 1-5'   # Re-run at 7:00 AM PDT as backup
  workflow_dispatch:           # Allow manual trigger from GitHub UI

jobs:
  update-data:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - name: Fetch all data
        env:
          FRED_API_KEY: ${{ secrets.FRED_API_KEY }}
        run: python scripts/fetch_data.py
      - name: Commit updated data files
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/
          git diff --staged --quiet || git commit -m "chore: daily data update $(date -u +%Y-%m-%d)"
          git push
```

**Setup required in GitHub:**
1. Go to repo Settings → Secrets and variables → Actions
2. Add secret: `FRED_API_KEY` = your FRED API key
3. Vercel auto-deploys on every push — no additional config needed

## 9. Key Constraints

1. **Do not change the UI design.** Colors, fonts, layout, component structure — all preserved exactly. Only the data values and their population mechanism changes.
2. **Verify before wiring.** Every API call must be tested and confirmed to return real data before it's connected to the UI. Print output during Phase 2 testing.
3. **Never expose the FRED API key in client-side JS.** The key lives only in the Python script (server-side / GitHub Actions). The browser only reads static JSON files.
4. **Semi-static fields get stale badges.** Any field in `static_overrides.json` older than 45 days gets a `⚠ STALE` visual indicator in the UI automatically.
5. **Graceful fallback.** If a JSON fetch fails, the UI shows the last known value with a "⚠ Data unavailable" indicator — it never crashes or shows blank cards.
